Architecture Approaches for Kalina Engineering Attendance System

Choose the backend architecture that fits your GoDaddy shared hosting

A

Vanilla PHP + Custom REST API

Recommended — Pure PHP, no framework overhead

  • Custom router + controller pattern (lightweight MVC)
  • JWT auth via firebase/php-jwt (single file include)
  • Direct MySQL with PDO (prepared statements)
  • Works on any shared hosting — no Composer SSH needed
  • Full control, no framework learning curve
  • ~15-20 PHP files for the entire API

Pros

  • Zero dependencies issues
  • Easy to deploy via FTP
  • Complete control over code
  • Fastest execution on shared hosting

Cons

  • More boilerplate to write
  • No built-in ORM or validation
  • Manual CORS, rate limiting
B

CodeIgniter 4 Framework

Lightweight PHP framework with REST support

  • Built-in routing, validation, ORM (Query Builder)
  • REST API via codeigniter-restserver
  • Can be installed without Composer (manual download)
  • Good documentation, active community
  • Built-in security features (CSRF, XSS filtering)

Pros

  • Faster development with built-in tools
  • Input validation out of the box
  • Query builder prevents SQL injection
  • Migration system for DB changes

Cons

  • Framework overhead (~5MB)
  • Learning curve if unfamiliar
  • Some features may need Composer
  • Heavier than needed for a REST API
C

Slim Framework (Micro-framework)

Minimal PHP framework built specifically for APIs

  • Purpose-built for REST APIs
  • PSR-7 HTTP message interface
  • Middleware support (auth, CORS, rate limiting)
  • Requires Composer installation

Pros

  • Clean middleware architecture
  • Modern PHP patterns
  • Lightweight (~1MB)
  • Great for APIs specifically

Cons

  • Requires Composer (may need SSH on GoDaddy)
  • No ORM included
  • Smaller community than CI4
  • May not work on all shared hosting plans

Why I recommend Option A (Vanilla PHP)

For GoDaddy shared hosting, vanilla PHP gives you zero deployment friction. You upload files via FTP/File Manager and it works. No Composer, no SSH, no server configuration. The API is small enough (~15-20 endpoints) that a framework is overkill. We'll structure it with a clean MVC-like pattern so it's maintainable without the framework overhead.
